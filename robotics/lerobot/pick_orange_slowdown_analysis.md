# SFT 微调后"成功但变慢"现象分析（pick_orange，60 集）

> 现象：60 集 demo，绝大多数 episode < 30s；SFT 微调后成功率 70–80%，但**单次完成时长拉长到 60–90s（2–3 倍）**。
> 这种「能干、但慢」是 BC/SFT 类 VLA 微调中**最经典**的失效模式之一，原因通常不是单点，而是几类机制叠加。下面按出现概率从高到低排列。

---

## 1️⃣ Action 分布平均化（mode covering） —— 最可能的主因

### 机制
- 60 个 demo 不可能轨迹完全一致：抓取角度、接近路径、夹爪闭合时机都有差异
- 用 L2 / flow matching / MSE 这类回归损失训练时，模型对多模态动作分布**取均值**
- 多个不同方向 / 不同速度的演示求均值后 → **速度向量幅值被压缩** → 机器人走得更慢
- 即使位置仍能到达，每一步动作都比 demo 慢一截

### 诊断方法
```python
# 对比 SFT 模型预测动作幅值 vs demo 动作幅值
demo_speed     = np.linalg.norm(demo_actions[:, :3], axis=-1)   # 末端线速度
policy_speed   = np.linalg.norm(policy_actions[:, :3], axis=-1)
print(demo_speed.mean(), policy_speed.mean())
```
若 `policy_speed / demo_speed < 0.5` → 高度怀疑是 mode averaging。

### 缓解
- 用 flow matching / diffusion 代替直接回归（π₀ / π₀.₅ 路线就是为此）
- 给动作幅值加 **temperature** 或 **action scale 后处理**：`a = a_pred * gain` (gain ∈ [1.2, 2.0])
- 用 **action chunking + 大 horizon** 让动作"承诺"较远未来
- 在数据层让 demo **速度风格更一致**（重新采几集快速演示，旧的慢演示剔除）

---

## 2️⃣ 复合误差导致的"绕行 / 回退"循环

### 机制
- BC 的经典短板：策略偏离演示分布 → 进入 OOD 状态 → 又要"绕回"
- 60 集覆盖率极低，几乎肯定会有 OOD
- 表现：靠近橘子 → 偏一点 → 退回来 → 再靠近 → 再偏 …… **同一段轨迹反复出现**

### 诊断方法
看 rollout 视频里 **末端轨迹**：
- 是单调向目标？→ 不是这个原因
- 是来回锯齿/盘旋？→ 就是 compounding error

或对比 demo 与 rollout 的轨迹长度：
```
total_path_len(rollout) / total_path_len(demo)  > 1.5  → 大概率绕行
```

### 缓解
- 加 DAgger / interactive correction（人工干预若干次）
- 加 noise augmentation（在 demo state 上加噪声训练，提高鲁棒性）
- 加 action chunking + receding horizon，减少每一步的暴露偏差
- 数据增广：随机时序裁剪、随机起点

---

## 3️⃣ 数据量太小（60 集）→ 整体策略变保守

### 机制
- 60 集对单一任务勉强够，但**远不够覆盖物体/光照/姿态的方差**
- 训练损失最低点对应"小幅、缓慢、靠近 demo 状态分布中心"的行为
- 模型学到的是「贴着 demo 中心慢慢走」而不是「快速、果断」
- 这种保守在 BC 文献里叫 **policy uncertainty → reduced action magnitude**

### 诊断方法
- 看末端**接近目标**最后 10cm 的速度曲线：典型问题是速度提前下降，**距离还有 10cm 时已经爬行**
- 看夹爪闭合时机：是否在到位前**反复试探**

### 缓解
- 扩集数到 200+（最直接）
- 借鉴 π₀.₅ co-training：混入其他抓取任务/其他机器人数据
- 大幅 image augmentation（color jitter / crop / rotate）—— 不增量数据但增加视觉鲁棒性
- 用 pretrained VLA（π₀ / OpenVLA / RDT）做底座再微调，而非从头 SFT

---

## 4️⃣ 演示本身有"等待 / 思考"间隙被学进去了

### 机制
- 人 teleop 时常有微小停顿（思考下一步、等夹爪反馈）
- 60 集里每集若有 2–3s "无意义停顿"，模型会**学到"经常停下"**
- 推理时叠加 policy 自身的犹豫 → 停顿被放大

### 诊断方法
统计 demo 中动作幅值低于阈值的占比：
```python
low_action_ratio = (np.linalg.norm(demo_actions, axis=-1) < eps).mean()
# 若 > 10% 就要警惕
```
对比同样统计 rollout 中的比例 —— 通常 rollout 中比例会**膨胀 2–3 倍**。

### 缓解
- 数据清洗：剔除动作幅值 < ε 的帧（或合并成更短序列）
- 或对每集**速度归一化**（按 episode 长度等比拉伸时间轴重采样）
- 重新采集时让操作者**全程不停手**

---

## 5️⃣ 控制频率 / action chunk 配置不当

### 机制
- demo 采样 30Hz，推理时模型实际控制频率只有 5–10Hz？→ 同样动作 chunk 走完物理时间变长
- action chunk H 太长 + 没用 receding horizon：一旦预测了一段慢动作就必须走完
- 或者推理时每步只取 chunk 的第一个 action（temporal aggregation 没开）→ 重复计算 + 抖动

### 诊断方法
- 测 policy inference latency：单次 forward 时间 × chunk 步数 = 应有控制周期
- 看 robot log 实际下发动作的频率

### 缓解
- 把控制频率提到 demo 频率以上
- 用 action chunking 的 **temporal ensemble**（ACT 风格）
- 检查是否启用 **flow matching 的 10 步去噪**（π₀.₅ 用 10 步，再多就慢了）

---

## 6️⃣ 夹爪 / 状态量没归一化 → 模型在某些维度过保守

### 机制
- 不同 episode 起始位姿差异大，若没做合适归一化，模型在远离 demo 起点位置时输出小幅动作
- 夹爪宽度若用绝对值，新环境一旦不同 → 整段都偏慢
- π₀.₅ 用 **per-dataset 1%/99% 分位数归一化**就是为此

### 诊断方法
看动作直方图：哪些维度的预测值集中在 0 附近 → 那些维度被压扁了

### 缓解
- 用分位数归一化（[-1,1] 范围）而不是 min-max 或不归一化
- 状态量做相对化（相对于物体 / 相对于起点）

---

## 7️⃣ 任务终止判定 / 抓后阶段"磨蹭"

### 机制
- "完成" 阶段（抓住橘子之后做什么？放下？保持？）在 demo 里可能不一致
- 模型对**结束条件**不确定 → 抓住后还在微调位姿、来回试探
- 60–90s 中有 20–40s 可能就花在"抓完之后干嘛"上

### 诊断方法
分段计时：
- T1：开始 → 接触橘子
- T2：接触 → 完全闭合
- T3：闭合 → 任务被判定为成功

看 T3 是不是主要膨胀的部分。

### 缓解
- 加显式 termination token（学一个 done 标签）
- 数据上让所有 demo "抓住即停"，删除尾部多余帧
- 推理上加成功检测器，触发即停

---

## 推荐的排查顺序（最低成本先做）

1. **看 5 段 rollout 视频**，分别打 3 段时间戳：接近 / 抓取 / 抓后
   → 判断时间膨胀主要在哪段
2. **统计 rollout 中"动作幅值 < ε"的帧占比**
   → 区分"走得慢"还是"经常停"
3. **对比 demo / rollout 末端路径长度**
   → 区分"走得慢"还是"走得绕"
4. **检查动作归一化方式 + 控制频率**
   → 排除工程配置问题
5. **采集另外 10–20 集"快速版本" demo 重训**
   → 验证是否单纯 mode averaging

---

## 概率最高的组合（个人判断）

> 60 集 + SFT + 速度膨胀 2–3 倍，最常见组合是：
> **#1 mode averaging（演示速度风格不一）**
> + **#3 数据量太小，策略保守**
> + **#7 抓后阶段磨蹭**

这三者叠加可以独立解释 2–3 倍时长膨胀，**不需要假设有 bug**。

最有性价比的两步缓解：
1. 数据层：**重新采 20 集"快速、无停顿"的演示**，与现有 60 集混合（或替换最慢的 20 集）
2. 推理层：在动作输出乘 1.3–1.5 的 **gain**，先看时长是否能压回 40s 内、成功率是否还在 70%
