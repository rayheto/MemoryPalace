# EtherCAT 硬件基础：为什么必须有专用芯片

## EtherCAT 不是纯软件协议

EtherCAT 有一个核心特性叫**飞速处理（on-the-fly processing）**：

```
主站发出一帧 → 从站1读走自己的数据、插入自己的数据 → 从站2同样操作 → ... → 帧返回主站
```

这整个过程要求每个从站的处理延迟**小于 1 微秒**。

Linux 进程的调度延迟是几十到几百微秒，根本无法在软件里实现这个速度。
因此必须用专用硬件芯片（ESC）在**电路层面**直接处理以太网帧。

---

## ESC 芯片的职责

ESC（EtherCAT Slave Controller）芯片负责：

| 功能 | 说明 |
|------|------|
| 帧的硬件转发 | 收到帧后纳秒级处理并转发给下一个从站 |
| DC 时钟硬件同步 | 维护本地时钟、计算误差、自动补偿，更新 `0x092C` 寄存器 |
| PDO 数据搬运 | 把帧里的数据搬到内部 DPRAM，供从站 CPU 读取 |
| 看门狗 | 通信中断时自动让输出进入安全状态 |

这些都是**纯硬件逻辑**（FPGA 或 ASIC 实现），软件无法替代。

---

## 从站完整硬件结构

```
以太网口 IN ──► ESC 芯片 ──► 以太网口 OUT（级联下一从站）
                  │
               DPRAM（双口 RAM）
                  │
              从站 MCU（运行用户逻辑，通过 DPRAM 读写过程数据）
```

从站软件（如 SOES）运行在 MCU 上，通过 DPRAM 与 ESC 通信，**不直接处理以太网帧**。

---

## 常见 ESC 芯片

| 芯片 | 厂商 | 特点 |
|------|------|------|
| ET1100 / ET1200 | Beckhoff | 最常见，工业标准 |
| LAN9252 / LAN9253 | Microchip | 集成度高，带 SPI/并口接口 |
| GD32E503 | 兆易创新 | 国产，集成 ARM Cortex-M 核 |
| XMC4800 | Infineon | 自带 EtherCAT 从站控制器的 MCU |

## EtherCAT 从站状态机

从站有 4 个状态，按顺序递进：

```
INIT → PRE-OP → SAFE-OP → OP
```

| 状态 | 输入数据（从站→主站） | 输出数据（主站→从站） | 说明 |
|------|----------------------|----------------------|------|
| INIT | 不传输 | 不传输 | 初始化，配置邮箱 |
| PRE-OP | 不传输 | 不传输 | 配置 PDO 映射、DC 参数 |
| SAFE-OP | **正常传输** | **被忽略** | 只读模式，验证通信链路和时钟 |
| OP | **正常传输** | **正常执行** | 完全运行状态 |

### SAFE-OP 的意义

SAFE-OP 是"只读模式"：从站上报自己的状态，但拒绝执行主站的控制命令。
用途：在正式运行前验证 PDO 通信正常、DC 时钟已收敛，避免带着错误时序直接驱动电机。

---

## System Time Difference 寄存器（0x092C~0x092F）

- **类型**：ESC 芯片内部硬件寄存器，芯片自动维护，无需软件手动写入
- **内容**：从站本地时钟与总线参考时钟的差值，单位纳秒，有符号 32 位整数
- **更新时机**：每次收到含 DC 时间的帧时，ESC 硬件自动刷新

```
ecx_configdc() 写入 DCSYSOFFSET + DCSYSDELAY
        ↓
ESC 硬件用这两个值补偿本地时钟
        ↓
补偿后的残余误差 → 硬件自动写入 0x092C（System Time Difference）
        ↓
主站读此寄存器评估同步精度，低于 1000（1 微秒）为合格
```

主站读取方式：
```c
int32 diff;
ecx_FPRD(&ctx.port, slaveh, 0x092C, sizeof(diff), &diff, EC_TIMEOUTRET);
// diff < 1000 ns 说明 DC 同步良好
```

---

## 一句话总结

**EtherCAT 从站 = ESC 硬件芯片（必须）+ MCU 软件（跑业务逻辑）。**
没有 ESC 芯片，再好的软件也无法成为合规的 EtherCAT 从站。
